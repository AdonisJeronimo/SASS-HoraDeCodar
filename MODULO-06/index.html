Módulo 6

Diretiva

A diretiva @use é uma funcionalidade introduzida no SASS (a partir do final de 2019) para importar módulos CSS e criar namespaces. Essa diretiva, esta substituindo gradativamente a diretiva @import nas versões mais recentes do SASS. Aqui estão algumas razões pelas quais é recomendado usar @use em vez de @import:

  1. Escopo mais seguro e controle de nomenclatura: Com o @use, os estilos, variáveis, mixins e funções importados são encapsulados em um namespace. Isso ajuda a evitar conflitos de nomeação e torna mais fácil entender de onde vêm as declarações importadas. O @use promove um escopo mais seguro para evitar sobrescrever acidentalmente estilos ou variáveis em diferentes partes do código.
  2. Melhor organização e modularidade: O @use incentiva uma abordagem mais modular para a organização do código CSS. Você pode importar apenas os módulos específicos que precisa, em vez de importar um arquivo inteiro. Isso melhora a manutenção e a legibilidade do código, pois você pode ver facilmente quais módulos estão sendo usados em cada arquivo.
  3. Resolução de dependências mais robusta: O @use resolve automaticamente as dependências entre módulos, evitando problemas de ordem de importação. Isso significa que você não precisa se preocupar com a ordem em que os módulos são importados, pois o SASS cuida disso automaticamente.
  4. Carregamento assíncrono e parcial: O @use carrega módulos assincronamente por padrão, o que pode melhorar o desempenho do compilador SASS. Além disso, o @use permite a importação parcial, o que significa que você pode importar apenas as partes específicas de um módulo que precisa, reduzindo a quantidade de CSS gerado.
  5. Evitar duplicação e problemas de cascata: Com o @use, os estilos importados não são incluídos novamente se você já os importou em um arquivo anterior. Isso evita a duplicação de estilos e problemas de cascata que podem ocorrer ao importar o mesmo arquivo várias vezes.
  6. Melhor suporte para bibliotecas externas: O @use fornece suporte nativo para bibliotecas externas, permitindo importar módulos diretamente de bibliotecas instaladas via gerenciador de pacotes. Isso facilita a integração de bibliotecas externas em seu projeto.
  7. Depreciação futura do @import: O @import foi depreciado no SASS e está sendo gradualmente removido em versões futuras. Portanto, é uma prática recomendada migrar para o @use para garantir a compatibilidade e se beneficiar das melhorias e recursos adicionais.

Instalando o LiveServer

O Live Server é uma extensão do Visual Studio Code que permite executar um servidor de desenvolvimento local para projetos web. Com essa extensão, você pode facilmente abrir um arquivo HTML, CSS, JavaScript ou qualquer outro arquivo da web e visualizá-lo no navegador em tempo real, sem precisar recarregar a página manualmente a cada alteração.

Arquitetura SMACSS

SMACSS (Scalable and Modular Architecture for CSS) é uma metodologia de organização e estruturação de código CSS desenvolvida por Jonathan Snook. Ela é projetada para tornar o código CSS mais modular, escalável e fácil de manter.

A estrutura SMACSS é baseada em cinco categorias principais para organizar os estilos:

  1. Base: Essa categoria contém estilos de nível mais baixo, como definições de estilos para elementos HTML (por exemplo, estilos para h1, p, a, etc.) e regras gerais de formatação que se aplicam a todo o projeto. Os estilos da categoria Base geralmente não têm classes ou seletores específicos, pois são aplicados globalmente.
  2. Layout: Aqui, você define estilos para estruturas de layout, como contêineres, grades, áreas de cabeçalho e rodapé, barras laterais etc. Esses estilos ajudam a estabelecer a estrutura visual do seu projeto e costumam ter classes específicas para identificar os elementos.
  3. Module: Essa categoria abrange os componentes individuais do seu projeto, como menus, botões, formulários, caixas de diálogo etc. Cada módulo deve ser encapsulado em uma classe específica para torná-lo independente e reutilizável em todo o projeto.
  4. State: Essa categoria inclui estilos para estados específicos de um elemento, como hovers, ativos, focos, desabilitados etc. Os estilos de estado são aplicados aos elementos com base em eventos ou interações do usuário.
  5. Theme: Essa categoria aborda a personalização visual do projeto para diferentes temas ou variações. Ela permite que você defina estilos específicos para diferentes aparências do seu projeto, como tema claro, tema escuro etc.

A estrutura SMACSS encoraja a separação clara de responsabilidades entre as categorias, tornando mais fácil a manutenção e o ajuste fino do código CSS. Ela também promove o uso de classes e seletores específicos para evitar conflitos e melhorar a reutilização do código.

No entanto, é importante lembrar que a estrutura SMACSS é uma abordagem flexível e pode ser adaptada às necessidades do projeto.

Arquivo principal da aplicação

O arquivo principal vai vai apenas realizar importações, não vamos adicionar estilos a ele. Podemos chamá-lo de main.sass ou app.sass.

As pastas de outros módulos iniciam com números, por exemplo: 0-plugins.

Estrutura de pastas

  • plugins: Frameworks ou códigos de terceiros.
  • base: Pasta para estilos básicos e de nível mais baixo, como reset de estilos, definições de tipografia e utilitários.
  • layouts: Pasta para estilos de layout, como cabeçalho, rodapé, barras laterais, etc.
  • modules: componesntes das páginas.
  • themes: Pasta para estilos relacionados a diferentes temas ou variações do design.

Arquivo principal dos subdiretórios

  • O arquivo principal do subdiretorio também só irá importar arquivos.
  • Isso vai diminuir os imports no nosso arquivo principal.
  • Organizando ainda mais a estrutura.
  • Nomeclatura para 1-base => base.sass

Base

  • Vamos agora criar alguns estilos de base.
  • Podemos estilizar tags como: body, a , p, etc.
  • Não utilizamos classes nem IDs
  • A ideia é pré-estilizar nosso HTML.
  • Podemos estilizar elementos em arquivos separdos.

Layouts

  • Aqui podemos adicionar IDs e classes as regras
  • A ideia é estilizar componentes maiores, como cabeçalho da página(header).
  • Novamente podemos separar elemetnos em arquivos distintos

Modules

  • Aqui podemos adicionar IDs e classes as regras
  • Vamos colocar nossos componentes menores:
    • Cards;
    • Botões;
    • Alguns elementos na sidebar;
    • etc.
  • Podemos também separar cada um deles em um arquivo.
